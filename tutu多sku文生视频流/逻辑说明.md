# Multi-SKU 5s 版逻辑说明

## 1. 为什么要做这一版

`agent_fast` 已经验证了"A1 context → Seedance 文生视频 prompt → 文生视频"的跳过文生图路线。它的默认是 15 秒、单参考图、允许自然运镜但对切镜没有强约束。

在实际产出里出现两个共性问题：

1. **尺度失控**：不管 prompt 里怎么写"4cm""微缩""不要超过画面 X 比例"，Seedance 都可能把 TUTU 画得很大。测试多次仍然无效。
2. **切镜太多**：15 秒里 Seedance 倾向于做镜头切换、POV 切、推拉切，让视频看起来像剪辑出来的，不像一个连续的生活片段。

这一版本把这两个问题作为核心约束重做：

- 把时长降到 **5 秒**，更适合单连续镜头
- 强约束**一个连续镜头**，允许合理运镜，但不允许切镜
- 用 **xiangyu 的尺度控制方法**：不再用否定词，而是通过"道具和蘑菇TUTU 的具体大小对比"来间接锁定小尺度
- 多 SKU 随机化：7 种蘑菇TUTU 任选，测试同一个 prompt 在不同 SKU 上的稳定性
- 引入 **3 张参考图**：SKU 四视图 + 手脚 + 嘴巴，锁死角色关键细节（肢体末端形态、嘴形）

## 2. 尺度控制的逻辑（最关键）

### 2.1 已知失败的做法

```
不要超过画面三分之一。
不要超过画面一半。
蘑菇TUTU 是 4cm 微缩角色。
保持微缩体量。
```

这些都是**抽象指令**，模型不知道怎么把"三分之一"对应到实际像素，也不知道"微缩"到底多大。大量试验证明它们都不能稳定生效。

### 2.2 xiangyu 的做法（被采纳）

xiangyu 的 `examples-library.md` 里的高质量 prompt 从来不写"不要超过画面 X 比例"，而是**把周围物体和蘑菇的具体大小比例直接写进场景描述**，让模型通过物理世界类比去推断角色尺寸：

```
枕头约是秃秃身高的 3 倍宽，闹钟约是它身高的 1.5 倍。
订书机约是秃秃身高的 2 倍长，秃秃站在旁边只到订书机的一半高。
水坑约是秃秃身高的半径大，落叶比它半个身体还大。
油条比它整个身体还长。
```

物理比较比抽象画面比例更符合 Seedance 的生图逻辑。只要场景里的**可识别物体**（杯子、水果、书本、面包、袜子、订书机、枕头等）和蘑菇TUTU 之间建立了清楚的相对尺度，模型会自动把蘑菇TUTU 画成一个微缩角色。

### 2.3 本版的约束

system prompt 里强制要求：

1. `场景：` 段落里**至少有两处**"蘑菇TUTU 与具体物体的大小对比"
2. 关键物体**出场时必须带上大小对比**
3. 可以补一句"蘑菇TUTU 作为微缩尺度主体，身高只到画面高度的五分之一左右"作为辅助锚点（属于画面占比描述，但用"辅助锚点+物体对比"的组合比单独用画面比例更稳）
4. **禁止**单独使用 `4cm` / `微缩尺度` / `比例` 这种抽象名词
5. **禁止**写"不要超过画面三分之一"这种直接画面占比指令

Phase C 脚本里通过 `FORBIDDEN_SUBSTR_REPLACEMENTS` 对这些词做硬过滤。

## 3. 单镜头 + 合理运镜

### 3.1 为什么要禁切镜

5 秒视频本来就很短，如果再切镜，每个镜头只有 1-2 秒，Seedance 在这种子镜头上很难保持角色外观和物体位置稳定，会出现跳变、透视错位、角色瞬移等问题。

### 3.2 允许什么运镜

`镜头：` 段落可以从以下池子里选一到两个组合，但**整条是一个连续镜头**：

- 固定机位
- 桌面平拍
- 贴近地面低机位
- 轻微推近
- 轻微拉远
- 轻微侧移
- 轻微跟随
- 轻微俯仰
- 轻度手持呼吸感

### 3.3 禁止写什么

system prompt 和 Phase C sanitize 一起封掉：

- "切到 POV" / "切到俯视" / "切到近景"
- "镜头切换" / "切换镜头"
- "分镜"

sanitize 会在 Phase C 产出后把这些词替换成空串兜底，避免 Gemini 偶尔冒出分镜语言。

### 3.4 动作节奏

动作链**只用顺序词**串：首先 / 接着 / 随后 / 然后 / 紧接着 / 停顿一下 / 最后 等。**禁止**写时间段或时间码（不允许 `0-1.5s`、`1-3秒`、`第2秒`、`前两秒` 等）。

原因：

- 时间码会让模型把动作当成**分镜刻度**，在 5 秒里容易触发切镜/跳切
- 顺序词保留节奏感，但把时间分配交给模型自己流畅展开，更像一个连续镜头的自然叙述

参考一个合格的动作段落写法：

```
场景：首先蘑菇TUTU 站在桌面边缘，抬头看向比它整个身体还大一圈的面包片，豆豆眼圆圆地亮了一下；接着它踮起短腿，两只短手费力扒住面包边，身体微微后仰使劲拉动，面包在桌面上被拖出一道细小的碎屑痕迹；随后它停下来喘了一口气，蘑菇帽微微歪到一侧；最后它松开面包，软塌塌地坐回桌面，眯眼轻轻"嘟～"一声。
```

整段只有"首先/接着/随后/最后"这四个顺序词，没有任何时间数字，但节奏清楚。

## 4. Multi-SKU 随机化

`秃秃sku四视图/` 里有七种蘑菇TUTU 的四视图。这版希望**每条 prompt 随机用一种**，好处：

- 测试 prompt 的**SKU 稳定性**：同一个场景、不同 SKU，外观特征能不能都稳得住？
- 快速看哪些 SKU 容易出问题（例如某种带白点的花纹 Seedance 总是画错）
- 批量时视觉上更丰富，不会 50 条都是同一个蘑菇

### 4.1 随机时机：Phase C

SKU 编号在 Phase C 里随机，写到 JSONL 的 `sku_index` 字段；Phase D 只是忠实按编号取图片路径。

好处：

- 结果可审计，每条记录都写清楚用了哪个 SKU
- 可复现：`--seed` 控制随机种子
- Phase D 是纯粹的"取图 + 提交"，不需要再做随机决策

### 4.2 Prompt 里不写 SKU 颜色

Prompt 首段只说"图片1是蘑菇TUTU 的四视图"，不写"粉色帽""橙色伞盖"等具体长相词。原因：

- agent_fast 的经验是：写长相词容易被模型当成**描绘目标**而不是**参考目标**，导致画出错误混合体
- 让图片 1 本身去承担颜色/花纹信息，text 只做"用图片1 的形象"这种指令

## 5. 三张参考图的作用

| 图片 | 用途 | Prompt 里的引用 |
|------|------|---------------|
| 图片1 | SKU 四视图（随机 1/7） | 严格参考蘑菇TUTU 形象、毛绒质感、身体比例、帽子颜色 |
| 图片2 | 手和脚参考图（hand_foot.jpg） | 肢体末端的毛绒圆球形态和粉色小肉垫颜色 |
| 图片3 | 嘴巴参考图（mouth.jpg） | 嘴巴张开时的嘴形、嘴内颜色和粉红色嘴唇轮廓 |

三张图都传 `role: reference_image`。xiangyu 的实践已经证明 Seedance 可以同时接多张 reference_image（他用到 7 张），不会冲突。

### 5.1 为什么要单独提手脚和嘴巴

- **手脚**：蘑菇TUTU 的肢体末端是**毛绒圆球 + 粉色肉垫**，没有手指、没有爪子。这是角色外观的死线，Seedance 在没有细节参考时容易画出手指或爪子
- **嘴巴**：张嘴时里面是特定的嘴形、颜色和嘴唇轮廓，Seedance 在没有细节参考时容易画出牙齿或人类嘴形

用单独的细节参考图比把这些细节写进文字 prompt 更稳。

## 6. Prompt 格式：seedance_prompt_agent 风格

`seedance_prompt_agent` 的 `seedance_prompt_writer_system.md` 定义了一种段落格式：

```
<首段图片关系>

风格：...
镜头：...
场景：...
配乐/音效：...
约束：...
```

这个格式的优点：

- **结构清晰**：每段职责明确，Seedance 容易按段落解析
- **可验证**：脚本可以直接检查段落标签是否齐全
- **稳定**：相比自由长段落，出错率更低

本版的 `phase_c_multi_sku_t2v_prompts.py` 的 `validate_prompt_shape` 强制校验：

- 首段以"图片1是蘑菇TUTU的四视图"开头
- 出现 `图片1` `图片2` `图片3` 三个引用
- 出现 `风格：` `镜头：` `场景：` `配乐/音效：` `约束：` 五个段落标签

任何一项失败就 retry，最多 2 次。

## 7. 和 agent_fast 的代码关系

本目录下的 Python 代码完全自包含，不依赖 `agent_fast/` 下的任何脚本。具体区别：

- **Phase A**：`phase_a.py` 是本目录独立的版本。行为和 `agent_fast/phase_a.py` 一致，但：
  - 输出目录默认为 `tutu多sku文生视频流/outputs/phase_a/<timestamp>/`
  - manifest 里的下游状态提示改成本 pipeline
  - IP 基础资料（`daily_signals.json` / `personality.md` / `ip-constitution.md`）放在本目录 `ip_data/` 下作为独立副本
  - **所有 system prompt 抽到 md 文件**：`phase_a_system_prompt.md`（主 prompt，含 `{personality}` / `{constitution}` 占位符）和 `phase_a_pool_subprompts.md`（七个池子的子 prompt）。Python 代码从 md 加载，不再硬编码 prompt 字符串。
- **Phase B**：`phase_b_multi_sku_blueprints.py` 是本目录新增的确定性准备阶段（`agent_fast` 没有）。职责：
  - 读 Phase A 的 `phase_a_contexts.jsonl`
  - 对每条 context 用 `random.Random(seed)` 抽 1/7 SKU
  - resolve `sku/{i}.png` / `sku/hand_foot.jpg` / `sku/mouth.jpg` 三张图的绝对路径，检查文件存在
  - 从 `action_theme` 派生 16 字 `title`
  - 写 blueprint JSONL（嵌套 `context` 子字段）
  - **不调 LLM**，因此没有 system prompt
- **Phase C**：`phase_c_multi_sku_t2v_prompts.py` 只做一件事：把 Phase B 的 blueprint 交给 Gemini，出 5 秒 T2V prompt；system prompt 和 sanitize 规则都和 `agent_fast/phase_c_*` 不同
- **Phase D**：结构和 `agent_fast/phase_d_seedance_t2v_videos.py` 类似，但 `build_payload` 接收 **3 张参考图 URL** 而不是 1 张，`duration` 默认改为 5
- **SKU 参考图**：七种 SKU（`1.png` ~ `7.png`）+ `hand_foot.jpg` + `mouth.jpg` 都复制到本目录 `sku/` 下。Phase B / Phase D 直接读本地路径，不再依赖 `../秃秃sku四视图/`。Python 代码、IP 数据、参考图三者全部自包含

### 为什么把 SKU 随机单独拆成 Phase B

原先 Phase C 同时做两件事：（1）挑 SKU、查路径、写标题；（2）调 Gemini 生成 prompt。

拆开之后：

- **职责清晰**：Phase B 是纯确定性逻辑（seed 固定输出就固定），Phase C 是纯 LLM 调用
- **快速迭代 prompt 写法**：改了 `phase_c_multi_sku_t2v_system_prompt.md` 的规则，只需重跑 Phase C，SKU 分配保持不变，可以直接对比不同 prompt 风格在**同一批 SKU** 上的效果
- **快速迭代 SKU 策略**：换 `--seed` 或换 `SKU_COUNT` 只需重跑 Phase B，不动 Phase C，省 Gemini 调用开销
- **可观测**：`phase_b_multi_sku_blueprints.md` 直接列出每条 context 用了哪个 SKU，方便审计 SKU 分布
- **Phase C**：完全新写，读 A1 context + 随机 SKU → 调 Gemini，system prompt 和 sanitize 规则都不同；默认自动找本目录最近的 `outputs/phase_a/*/phase_a_contexts.jsonl`
- **Phase D**：结构和 `agent_fast/phase_d_seedance_t2v_videos.py` 类似，但 `build_payload` 接收 **3 张参考图 URL** 而不是 1 张，`duration` 默认改为 5

## 8. 推荐验证顺序

1. `phase_c_multi_sku_t2v_prompts.py --limit 3`：先看 prompt 文本，尺度比例是否写得具体，有没有切镜词
2. `phase_d_multi_sku_t2v_videos.py --limit 1`：dry-run 看 payload，3 张图是否都在
3. `phase_d_multi_sku_t2v_videos.py --limit 1 --execute --wait --download`：跑一条真视频看效果
4. 看视频里：
   - 是不是真的 5 秒
   - 是不是一个连续镜头没切
   - 蘑菇TUTU 相对周围物体是不是微缩尺度
   - 手脚是不是圆球（图片2 有没有起作用）
   - 张嘴是不是参考嘴巴图的嘴形（图片3 有没有起作用）
   - 随机到的 SKU 外观对得上图片1

## 9. 未来扩展

- 若 5 秒实测下来尺度控制仍偶有失手，可以把 `场景：` 段落里要求的"具体比例对比"从"至少两处"提高到"至少三处"
- 若 3 张参考图太多导致 Ark 任务变慢，可以尝试把手脚和嘴巴合并成一张"细节合图"
- 若某几个 SKU 特别稳定或特别不稳定，可以把 `pick_sku_index` 换成加权采样
- 如果要做更短（3 秒）或更长（8 秒）的版本，可以复用这套 system prompt，只调整"5 秒"相关的描述
